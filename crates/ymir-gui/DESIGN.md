# Ymir GUI design (settled, round 3)

This is the design brief for `ymir-gui`, the node editor and viewport for Ymir.
It is to the GUI what `CLAUDE.md` is to the engine: the locked decisions plus a
stepwise build plan. This revision incorporates the round 1 and round 2 reviews
(crate survey plus design feedback, then two factual and precision fixes). When
settled, this file lives at `ymir-gui/DESIGN.md`.

The two items previously flagged for the maintainer (defer the Fluent backend behind
the `tr(key)` seam; build only the cheap workspace seams in v1) are now confirmed and
moved into the locked decisions.

## Purpose and scope

Ymir's stated goal is a terrain tool that is pleasant to compose with, not a clone
of Gaea or World Machine. The engine (`ymir-core`, `ymir-nodes`) is built and
frozen in its essentials. The GUI is where that goal is actually realised, so the
emphasis here is on the composing experience: how nodes are found, wired, tuned, and
previewed, and how the workspace itself is arranged.

The GUI is mostly a matter of driving seams the engine already exposes, plus a small
number of genuine design decisions that this document settles.

## How this maps to the roadmap

Most of this work is Track C (GUI). Several pieces reach into other crates and should
be filed under the appropriate track, not buried in a GUI issue:

- A `NodeSpec` change (category id, tags, and display text becoming resolver keys) is
  a `ymir-core` edit. Core-touching, additive, do it early.
- An `Operator: Send + Sync` bound, an optional eval cancellation token, and a
  per-node stale/cached query are `ymir-core` edits driven by background eval. See
  "Impacts on ymir-core".
- Category definitions and node string catalogs land in `ymir-nodes`.
- The Fluent localization backend (i18n-embed) is a later cross-cutting concern shared
  by `ymir-gui` and `ymir-cli`, sitting behind the resolver seam.

## Working style

This inherits `CLAUDE.md` wholesale: small single-purpose steps, each ending
compiling, tested, clippy- and fmt-clean; a checkpoint for review before the next
step; no symptom-hiding shortcuts; one focused commit per step. Get a pixel on
screen as early as possible and grow from there. The maintainer hands over one step
at a time and the pace is respected even when later steps are visible.

## What the engine already decides

These are not open questions. The GUI consumes them and must not duplicate or
contradict them:

- The node palette is generated by iterating the registry (`inventory`), never a
  hand-kept list.
- Parameter widgets are built by introspecting `ParamSpec` (name, type, range,
  default): float to slider, int to drag value, bool to checkbox, enum to dropdown.
  No per-node widget code.
- The global seed is a field of `EvalRequest` (the thing the GUI constructs). The
  evaluator derives each node's seed from it and the node's `stable_id` and places
  that per-node derived seed in `EvalContext` (what operators receive). A single
  top-level control sets the request's seed, with a randomise affordance. There is no
  per-node seed parameter and so no per-node seed widget; the GUI's seed control
  writes to `EvalRequest`, never to `EvalContext`. See B-1 and F-2 in the reviews.
- Save and load store `type_id` plus params plus connections plus `stable_id`, and
  rebuild via the registry.
- Evaluation is pull-based and memoized, requested from a chosen endpoint, with a
  bounded cache. Erosion is resolution-dependent physics; a low-resolution preview
  is an approximation of the build, never an identical result.

The invariant holds in the GUI too: nothing asks "which node is this?" Everything
dispatches polymorphically or reads the node's own spec.

## Technology

- egui plus wgpu, as already chosen for `ymir-gui`.
- The node editor adopts `egui-snarl` (0.10.0, targeting egui 0.34) rather than
  hand-building one. Its `SnarlViewer` trait decides each node's title, pin count,
  and per-pin UI, exposes a `connect()` hook to accept or reject wire attempts, and
  provides a background context menu (the Houdini tab-menu gesture). The round 1
  survey confirmed all three are present. It is MIT/Apache licensed, which composes
  into GPL-3.0 without issue.
- Survey caution, snarl churn: egui-snarl is low-adoption (about 1k downloads a month)
  and still evolving (nine breaking changes across fifteen releases). Pin the version,
  budget for migration work on upgrades, and treat it as a dependency to track
  actively, not fire-and-forget. It remains the right choice over hand-rolling.
- Survey caution, egui lockstep: the whole GUI stack upgrades in egui-version
  lockstep. egui-snarl, eframe, egui-phosphor, and any future egui_tiles or egui_dock
  must agree on the same egui major. Pin egui centrally in the workspace and expect
  periodic synchronized bumps as a single coordinated step.

## Single source of truth and connection reconciliation

`ymir-core`'s graph is canonical. The snarl structure must not hold a second copy of
node data: the snarl node type is a small handle (the `stable_id`), so node data
lives only in core. The survey confirmed this is achievable.

Connections are the subtle part. Snarl owns its wire state, so "no second copy" is
not automatic for edges; it needs an explicit reconciliation policy. The chosen
policy: snarl is the editing surface, and its `connect()` and `disconnect()` hooks
validate each attempt (arity and the DAG rule) and apply the accepted change to
core. Core stays the canonical, persisted authority and the validity authority;
snarl's wires only ever reflect edges core has accepted. Re-deriving snarl's wires
from core on every change is the alternative; it is rejected as fighting snarl's
interaction model, but the decision should be confirmed by a small spike (below).

Node lifecycle follows the same canonical-core discipline as wires: adding a node
from the ribbon or the tab-menu creates the core node (assigning a new `stable_id`)
and hands snarl only the handle; deleting a node in snarl removes it from core and
cascades to its edges. Snarl never holds a node without a backing `stable_id`.

A spike before step 5 confirms this reconciliation is ergonomic in practice. The
single-source-of-truth claim rests on it, so it is verified before the canvas step
proper begins. When this document is decomposed, the spike is its own tracked
de-risking issue, a real "prove the assumption" task, never folded silently into the
canvas step.

## The pane model (the central seam)

Every region of the UI is a self-contained pane: a function or widget that draws into
a passed `egui::Ui` given the application state, with no knowledge of where it is
mounted. A `palette_ui`, a `params_ui`, a `canvas_ui`, a `viewport_ui`, a
`preview2d_ui`. This decoupling is the cheap, high-value seam, and it is built in v1.

Pane kinds self-register, the same pattern as operators and categories: a pane kind
has a stable id (`"canvas"`, `"viewport-3d"`, `"params"`, `"preview-2d"`,
`"ribbon"`), and adding a new pane kind later is one registration, touching no layout
code. This is also built in v1; it is cheap and it is what genuinely keeps "add a
pane later" a one-registration change.

## App shell and default layout

The shell is the first build target: the frame everything mounts into. The default
layout, top to bottom and left to right:

- A menu bar (File, Edit, View, Graph, Help) in a top panel.
- A ribbon: node-category tabs on the left, a node search field, and the global seed,
  resolution selector, and Cook action on the right. Beneath the active tab, a node
  toolbar shows that category's nodes as buttons.
- The node canvas and the 3D viewport, side by side (side-by-side suits wide monitors;
  the pane model makes switching to stacked a trivial change later).
- A right column holding the parameter inspector above a compact 2D preview.

There is no standing palette column (the ribbon replaces it) and no full-width bottom
strip (it wasted space on a small preview).

v1 implements this as fixed, resizable native egui panels: the cheapest path to
pixels, no extra dependencies, with resizing and clear pane locations for free. The
default layout is described in code against the pane-kind registry, so panes are
mounted by id rather than by inline panel calls.

## Forward compatibility: customizable workspaces

Requirement: nothing in the design may foreclose a customizable UI. The target world
is Houdini-style, where the user defines workspaces (named desktops) that specify
which panes exist and where they sit, and switches between them.

The load-bearing seams that keep this open, built in v1, are exactly two and they are
cheap: panes are self-contained mount-agnostic widgets, and they are identified by a
registered pane-kind id. With those in place, migrating to customizable workspaces
later is a contained change, not a rewrite, which is what "don't foreclose" requires.

Deferred to when docking is actually built (per review B-4): the serializable
workspace tree (splits and tabs whose leaves are pane-kind ids plus sizing) and the
swappable layout backend (native panels now, `egui_tiles` or `egui_dock` later).
Introducing those later is itself additive and touches no pane internals, so building
them in v1 would be speculative machinery. The maintainer should confirm this scope,
since it touches the customizable-UI requirement directly; the judgement here is that
the two v1 seams fully preserve it.

## Node access: tabs, toolbar, search

Three views over one registry, never bespoke per-node code:

- The category tabs and their node toolbars group registered nodes by category.
- The search field fuzzy-matches over node names and tags.
- A canvas-summoned add: snarl's background context menu opens the same search on the
  empty canvas (the Houdini tab-menu gesture).

All node and category labels resolve through the localization seam (below).

## The category system

Categories are open and programmatic, never a fixed enum the GUI branches on. They
self-register exactly like operators:

- A node references its category by id in `NodeSpec`, plus optional tags for search.
- A `CategoryDef` carries an id, an icon id, and a sort order. It carries no prose:
  the display name and description are resolver keys derived by convention from the id.
- The icon is stored as an id string (for example `"droplet"`); the GUI resolves it to
  a glyph from an icon font (egui-phosphor 0.12, MIT). Textures are reserved for the
  rare missing glyph. The data layer stays free of egui types.
- Graceful degradation, mirroring the soft-layer contracts: a node whose category id
  has no registered `CategoryDef` falls into a synthesized "Uncategorized" group rather
  than failing. Nothing gates.

Categories are flat for now. Hierarchy is not built and not foreclosed.

## Localization (seam now, framework later)

Decision: all display text resolves through a key-based seam from day one, and no
string literal ever lives in a node or category spec. This is the part that must be
early, because turning a literal into a key later would touch every node already
written, which is not an additive change.

What is deferred (per review A-1): the Fluent backend itself. v1 routes every label
through a `tr(key)` resolver whose initial implementation is a trivial English lookup.
Reaching the first testable node therefore does not require standing up a multi-crate
Fluent framework first. When localization becomes a real goal, i18n-embed with the
Fluent system and the `fl!()` macro slots in behind the resolver, with no change to
nodes or specs. i18n-embed (0.16.0) is excellent and low-risk; this is purely a
sequencing call. The maintainer chose "do it right" earlier, so this synthesis (keep
the seam and convention, defer the framework) is flagged for confirmation.

The convention, in force from v1: specs hold ids; display text is keyed by
convention (`node-<id>`, `node-<id>-desc`, `category-<id>`, `category-<id>-desc`, and
likewise for parameter labels and tooltips).

When the Fluent backend lands:

- Not one `.ftl` per node. A node's code stays one self-contained file; its strings
  are a few lines appended to a per-language catalog. Fluent merges a language's
  messages into one bundle, and the key convention keeps them collision-free.
- Node and category catalogs live in `ymir-nodes`, localized as a library. The
  rationale that `ymir-cli` also wants node names (for a future list-nodes command and
  for error messages) is anticipatory today; flagged as such per review B-5. It still
  holds as catalog-placement guidance.
- `ymir-gui` carries only its own chrome strings (menus, toolbars, window text).
- English is the fallback and must be complete; `fl!()` makes a missing or mistyped
  key a compile error, restoring the "a typo is a compile error" property the engine
  has for layer-name constants.

`ymir-core` carries no i18n dependency and no prose, in either phase. It holds only
ids and keys.

## Evaluation orchestration and feel

This is where "pleasant to compose with" actually lives.

- Evaluation runs off the UI thread. egui is immediate-mode and must stay at
  framerate, so the eval runs as a background job the UI polls. This requires the
  `Operator: Send + Sync` bound (see Impacts).
- Live preview at low resolution, debounced, while a parameter is dragged. The
  full-resolution build is explicit (the Cook action). Latest-wins has two meanings,
  and the doc means both should be available: discarding a superseded result is free;
  aborting a superseded in-flight eval (worthwhile for slow erosion) needs a cancel
  token threaded through eval. See Impacts.
- Resolution honesty. The preview pane labels its resolution and never implies it
  equals the build, because erosion is resolution-dependent. The target-resolution
  build is the source of truth.
- Failures surface as red nodes: a failing node is shown in error state with its
  message while the rest of the graph still evaluates. No panics reach the UI.
- The canvas shows what is stale versus cached, which needs a per-node validity query
  on the cache or evaluator (see Impacts).

## Viewports

- The 2D heightmap preview comes first. It is a texture blit and is available from
  nearly the first step. It picks which layer and which output port to show, not only
  `height` (per review B-2): watching the mask a mask-node emits, or a flow layer, is
  the payoff of in-situ testing.
- The 3D viewport is a later milestone: a wgpu paint callback drawing a height mesh
  with a camera and simple height- or slope-based shading. Until it lands, its pane is
  a labelled placeholder, and the 2D preview carries the visual feedback.

## Testing strategy

"Each step ends tested" means something specific for a GUI, since rendering is not
unit-testable like the engine. Covered by tests:

- Workspace and pane-layout description serialization (once it exists).
- Registry and category iteration, and category graceful degradation.
- The `ParamSpec`-to-widget mapping logic, kept as pure logic separate from rendering
  so it is testable.
- The snarl-to-core sync invariant: no node in snarl without a backing `stable_id`,
  and no wire in snarl without a corresponding accepted edge in core.

Eyeball-verified, not unit-tested: layout, spacing, and feel.

## State and persistence

Two distinct kinds of GUI state, kept separate:

- Per-project view state (node canvas positions and similar) lives in a sidecar file,
  not in the project format (per review C). This keeps the canonical project file free
  of GUI concerns and headless-portable, the same "GUI state is not engine state" line
  the rest of the doc draws.
- App-level workspaces: named layouts, independent of any project, serialized to app
  config when the customizable-workspace work is done.

## Impacts on ymir-core and ymir-nodes

Flagged explicitly so they are filed correctly and done deliberately:

- `ymir-core`, additive and early: `NodeSpec` gains a `category` id and optional
  `tags`, and its display name and description become convention-derived resolver keys
  rather than string literals.
- `ymir-core`, driven by background eval: an `Operator: Send + Sync` bound. Today's
  operators (`Fbm`, `ThermalErosion`, `ExportPng`) are stateless unit structs, so they
  already satisfy it trivially and the bound is a no-op for them now; it matters
  because `Box<dyn Operator>` will cross to the eval thread, so the guarantee must be
  explicit for every future operator. (`Field` and `Layer` are already `Send + Sync`.
  There is no rayon dependency yet; it appears only as future intent in doc comments,
  so it is not a current source of the bound.) Also driven by background eval: an
  optional cancellation token threaded through eval, for aborting a superseded slow
  preview rather than only discarding its result; and a per-node stale/cached validity
  query for the canvas indicators.
- `ymir-nodes`: gains the `CategoryDef` type and category self-registrations. When the
  Fluent backend lands, also the i18n-embed dependency and the `.ftl` catalogs.
- `ymir-cli`: anticipated to read node names through the same catalogs (no list-nodes
  command exists yet).

## Build plan (MVP)

Each step is reviewable and runnable, done one at a time with a checkpoint. Pixels as
early as possible. Re-sequenced so reaching a testable node is not gated on the
localization framework (per review A-1).

1. App shell. An egui plus wgpu window, the menu bar, and the default layout as empty
   fixed native panels, with a hardcoded `Field` rendered in the 2D preview. Runnable
   window, no interactivity.
2. Pane model. The pane-kind registry and pane-as-function decoupling; the default
   layout described in code against the registry and mounted in fixed native panels.
   (The serializable workspace tree and swappable backend are deferred to the docking
   work.)
3. NodeSpec additive change. In `ymir-core`, add the `category` id and `tags`, and
   route display name and description through the `tr(key)` resolver, whose v1
   implementation is a trivial English lookup. No Fluent yet. Add the category
   self-registrations and `CategoryDef` in `ymir-nodes`.
4. Ribbon and palette. Category tabs, per-tab node toolbar, and node search, generated
   from the node and category registries, labelled via the resolver. The global seed,
   resolution, and Cook controls live here. Adding a node from the ribbon places it in
   the graph.
5. Canvas and parameters. Preceded by the snarl-as-pure-view reconciliation spike. The
   egui-snarl canvas renders nodes from `NodeSpec`; `connect()` and `disconnect()`
   validate arity and the DAG rule and apply to the canonical core graph; the parameter
   inspector is built from `ParamSpec` for the selected node. Add the
   `Operator: Send + Sync` bound here if not already done.
6. Evaluation and preview. Background eval, endpoint selection, debounced latest-wins
   live preview into the 2D pane with the layer and output-port picker, red-node
   errors, stale and cached indicators. Optionally the cancellation token.
7. 3D viewport. The wgpu paint callback, height mesh, camera, simple shading.

Later, beyond MVP: the Fluent localization backend behind the resolver; the docking or
tiling layout backend with user-editable and saveable workspaces; and save and load of
the graph plus sidecar view state. None are built in the MVP, but the seams above keep
them contained.

## Decisions locked

- egui plus wgpu; egui-snarl (pinned, tracked) for the node editor, with the core
  graph canonical and an explicit connect()/disconnect() reconciliation policy.
- egui pinned centrally in the workspace; the GUI stack bumps in lockstep.
- Fixed native panels for v1; no docking crate yet.
- Panes as self-contained, registered, mount-agnostic widgets: the two v1 seams that
  keep customizable workspaces open. The serializable workspace tree and swappable
  layout backend are deferred to the docking work (confirmed round 2: genuinely
  additive later, so not built now).
- Ribbon (category tabs plus per-tab node toolbar plus search) instead of a palette
  column; no bottom strip; global seed control in the ribbon; 2D preview (with a
  layer and output-port picker) under the parameter pane; 3D in its own pane;
  canvas and viewport side by side.
- Open, self-registered categories; `NodeSpec` carries a category id and tags.
- All display text resolves through a key-based seam from v1; the Fluent backend
  (i18n-embed 0.16) is deferred behind that resolver (confirmed round 2: keeps node
  testing off the i18n critical path, and slots in cleanly later).
- egui-phosphor for icons; textures only for missing glyphs.
- Per-project view state in a sidecar file; one toolbar in v1.
- 2D preview before 3D viewport; background eval; resolution-honest previews.

## Resolved since round 1

- Localization sequencing: key-seam and convention in v1, Fluent backend deferred
  behind the resolver. Confirmed.
- v1 workspace scope: pane decoupling and pane-kind registry only; serializable
  workspace tree and swappable backend deferred to the docking work. Confirmed.
- Canvas and viewport are side by side, with stacked still available later at low cost
  if it proves wrong on your monitors.
