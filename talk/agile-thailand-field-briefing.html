§01
The bet
Everything is data

One type flows on every edge

The whole architecture rests on a single wager: if every wire carries the same universal type, the engine never needs to know what any node does, and the user is never forced into a fixed build order.

That type is a Field: a grid resolution, normalized world-space region bounds, a map of named scalar layers (height, mask, flow, water, sediment…), and a small bag of scalar globals (seed, world bounds, vertical scale). Layers are Arc-shared, so passing a field through untouched is nearly free.

The discipline that keeps it a terrain tool rather than a general dataflow engine: Ymir adopts Houdini's attribute philosophy but not its element model. There is exactly one element — the 2D grid — plus globals. No points, no vertices, no primitives. Holding that line against scope creep is an explicit, recurring instruction.

Nodes declare the layers they want, degrade gracefully when one is absent, and never gate a connection. That is why any node is insertable anywhere.

§03
Structure
Architecture

The compiler enforces the invariant

The central rule: nothing may ask "which node is this?" Everything either dispatches polymorphically or reads the node's own schema. Adding a node touches only its own new file.

The dependency arrow does the work

Concrete nodes live in ymir-nodes; the engine lives in ymir-core, and the arrow points nodes → core. So the engine literally cannot name a concrete node — the invariant is enforced by the borrow checker, not by discipline. Good architecture is guardrails the builder can't step over.

The Operator trait

trait Operator {
  fn spec(&self) -> NodeSpec;                // identity, ports, params
  fn eval(&self, inputs, params, ctx)
        -> Result<Vec<Field>>;             // one field per output
  fn content_hash(&self) -> Option<u64>;    // for containers
  fn nested(&self) -> Option<&Graph>;       // subgraph nesting
}

Inputs are an ordered, named list from day one — required by index, optional via inputs.optional(i) — so multi-input blends and optional masks are first-class, never retrofitted. Failures are values, not panics: a red node is reported while the rest of the graph still evaluates.

Enforced in four independent places

Evaluate

Dispatch through dyn Operator::eval — core has no concrete type to match on.

Palette

Generated by iterating the inventory registry — no hand-kept list.

Params UI

Built by introspecting ParamSpec — zero per-node widget code.

Save / load

Stores type_id + params, rebuilds via the registry — no central enum.

A node's kind (generator / modifier / endpoint) is derived from arity, not stored: no inputs ⇒ generator, no outputs ⇒ endpoint. "Generators only at the head" enforces itself, because a generator has no input socket to wire.

§04
Evaluation
The evaluator & cache

Pull-based, memoized, three tiers deep

Validate the graph is a DAG, evaluate from the requested endpoint, recompute only downstream of a change. A full-resolution field per node would exhaust memory, so the cache is bounded by policy, not naïve.

Hot

The active evaluation path, in memory.

Warm · RAM

A small LRU for reuse across edits.

Warm · disk

Content-addressed blobs, keyed by computation — survive restarts, shared across projects.

Every cache operation is best-effort: a corrupt file or a full disk degrades to a miss, never a panic. A node's key is its type_id, a canonical param hash, its upstream input hashes, and the context fields it depends on — content-hashing everything is what makes the cache correct rather than merely fast.

§05
Contract
Determinism

A means to repeatability, not an end

The contract is stated in rungs — and refusing to over-engineer the top rung is itself the design decision.

  • Same machine — required, byte-identical. Same seed, graph, and machine yields the same output every time. The memo cache depends on it: keys are content hashes, so any run-to-run drift would make every downstream node miss constantly. No output may depend on hash-map order, the clock, a thread id, or address layout.
  • Across machines — visual equivalence. A shared project regenerates terrain that looks the same; a few last-bit differences in an order-sensitive parallel reduction are accepted rather than contorting the algorithm to force bit-identity nobody asked for.
Nobody chooses the tool because it is bit-exact across machines. They choose it because it reliably regenerates great terrain. Determinism serves that; it is not the trophy.

Per-cell pure functions (noise, shapes) write each cell independently, so they stay byte-identical regardless of thread count for free. Only order-sensitive scatter — thermal erosion adding into shared neighbours — relaxes.

Erosion is physics, not a sampled function

One more axis, and it is deliberate: iterative erosion is resolution-dependent. Run it at preview resolution and at build resolution and you get genuinely different terrain, not a rescaled copy — so the fast preview is a faithful approximation of the build, never a promise of pixel-identity to it. Its strength and scale are expressed in world units and its pass counts scale with cell size, so the preview stays representative rather than misleading. The target-resolution build is the source of truth.

§07
Expression
A tiny bytecode VM

The Expression node runs its own language

An expression is parsed once into flat bytecode, then a stack machine evaluates it per cell — no AST pointer-chasing, no per-cell allocation — so it stays fast over millions of cells and parallelizes trivially.

expr · per-cell stack machine
// height ramped by a warped radial falloff
lerp(h, smoothstep(0.2, 0.8, 1.0 - sqrt(x*x + y*y)),
     clamp(slope, 0.0, 1.0))

Built-ins: abs ceil floor sign sqrt exp ln pow min max clamp lerp sin cos tan step smoothstep select, constants pi e tau, and variables for cell coordinates and input layers. It is an expression language, not a script — branching is select(cond, a, b). An unknown identifier is a compile error, so a typo is reported, never silently zero.

Hand-rolled on purpose: the function set and numeric behaviour stay fully under control and byte-stable, with no dependency liability. Off-the-shelf evaluators are tree-walkers — the slow per-cell path.

Interface
The desktop application

egui + wgpu, and a self-referential preview

The GUI is the largest crate for a reason: a terrain tool lives or dies on how it feels to compose with. The node canvas is rendered as a pure view over the core graph — presentation and truth kept cleanly apart.

Node canvas

egui-snarl as a view over the authoritative core graph. The wiring is presentation; the graph is truth.

3D viewport

Real wgpu rendering inside an egui pane — terrain mesh, lighting, vertical-exaggeration slider.

Live thumbnails

Every node shows its own heightmap preview — cached, debounced, GPU-upload budgeted so it never hitches.

Curve editor

A visual transfer-curve widget, because shaping needs a widget, not bare sliders.

Subgraph library

Saved subgraphs as git-friendly, version-stamped JSON files — browse, insert, document.

2D preview

Off-thread evaluation with an output picker — inspect any tapped output of any node.

Full-res Build

Evaluates selected endpoints at target resolution — the source of truth behind the live preview.

Project format

One versioned, git-diffable JSON document. Undo/redo as whole-session snapshots.

§09
The method
Built with Claude Code

An agent under an enforced contract

Ymir was not just built with an AI assistant — it was built under an explicit, mechanically enforced methodology a human team could adopt wholesale. The claim to test on stage: with the right guardrails, the output holds up to a demanding reviewer, and the git history reads as a deliberate build narrative.

CLAUDE.md as a constitution

A ~250-line project brief the agent reads before every change: the vision, the data model, the invariants, the determinism contract, the quality bar, and the working style. Not documentation written after — the operating contract, written first.

Incremental, checkpoint-gated, one commit per step

Small single-purpose steps. Every step ends compiling, tested, clippy-clean, and runnable — the tree is never broken between steps. State the plan before, the change after, then stop for review before the next. The result is 252 commits you can scroll as a story, the opposite of a five-thousand-line dump.

No shortcuts — fix causes, not symptoms

The strongest rule, and enforced by machine, not vibes. A scanner recognizes symptom-hiding shapes and runs at two gates — a git pre-commit hook and an agent Stop hook, so the assistant cannot even end its turn having introduced one:

  • timers / sleeps to dodge a race
  • unwrap · expect · panic on expected conditions
  • #[allow] to mute a lint
  • todo! · unimplemented! · stubs
  • tests that assert nothing or are ignored
  • hardcoding a value to go green
  • swallowed errors (let _ =, .ok())
  • weakening a test to pass
Rule
CLAUDE.md defines forbidden shapes + an inline // shortcut-ok escape hatch, rare and visible.
Scan
check-shortcuts.sh greps the diff for those shapes.
Gate A
Pre-commit hook blocks the commit.
Gate B
Stop hook blocks the agent from ending its turn.

Tests, design docs, and a memory that compounds

  • 571 tests written with each step — unit per operator, property tests for invariants, golden snapshots that fail on any silent output change.
  • 13 design documents — anything non-trivial gets a written design and a checkpoint before code. Extension from clean seams, never speculative abstraction.
  • Persistent file-based memory — decisions and preferences carried across sessions, so the working relationship compounds instead of resetting each morning.
§10
Takeaways
Narrative hooks

Where the two threads meet

The through-line for the talk: the properties that make Ymir extensible for its users are the same ones that make it buildable by an agent.

Extensibility, twice

"Everything is data" means a new feature is one new file against a stable contract — extensibility for the user and for the builder are the same property.

Make the wrong thing not compile

The engine can't name a node, so it can't violate the additive invariant. Design so the mistake fails to build.

Guardrails > vibes

A Stop hook that refuses to end a turn on an unwrap is a memorable demo — and a transferable practice.

The human curates

An eager generator over-generalizes; the maintainer's job became saying no — encoded, so it holds.
